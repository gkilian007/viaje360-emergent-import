## ÔÜá´ŞĆ Issues Found

### ISSUE 1 ÔÇö `user_subscriptions`: Missing user-level READ policy
**Severity:** HIGH  
**Current state:** Only `service_role_all` policy exists. Authenticated users **cannot read their own subscription status** via the client SDK.  
**Impact:** Frontend subscription checks that use the anon/auth key directly will fail. Only works if all reads go through service_role (server-side API routes). 

> Ôťů Mitigated in practice: subscription checks are done server-side via `/api/access` route. But it's a footgun for direct client usage.

**Fix:** Add `auth_own` policy allowing users to read/update their own subscription.

### ISSUE 2 ÔÇö `destination_purchases`: Missing user-level READ policy
**Severity:** HIGH  
**Same pattern as user_subscriptions.** Users can't query their own purchases client-side.

### ISSUE 3 ÔÇö `destination_trials`: Missing user-level READ policy
**Severity:** HIGH  
**Same pattern.** Users can't read their own trial status client-side.

### ISSUE 4 ÔÇö `trip_day_activity_feedback`: Missing user-level policy
**Severity:** MEDIUM  
**Current state:** Only `service_role_all`. Authenticated users cannot read/write feedback client-side.  
**Fix:** Add policy scoped to `trip_id IN (user trips)` or a `user_id` column if it exists.

### ISSUE 5 ÔÇö `activity_events` table not found
**Severity:** LOW  
The task references `activity_events` but the actual table is `trip_activity_events`. No action needed ÔÇö just a naming discrepancy in documentation.

## Recommended Fixes

Migration file: `supabase/migrations/20260327000001_rls_auth_policies_fix.sql`

```sql
-- user_subscriptions: allow authenticated users to read/manage their own
create policy "auth_own_read" on public.user_subscriptions
  for select to authenticated
  using (auth.uid() = user_id);

create policy "auth_own_insert" on public.user_subscriptions
  for insert to authenticated
  with check (auth.uid() = user_id);

-- destination_purchases
create policy "auth_own_read" on public.destination_purchases
  for select to authenticated
  using (auth.uid() = user_id);

create policy "auth_own_insert" on public.destination_purchases
  for insert to authenticated
  with check (auth.uid() = user_id);

-- destination_trials
create policy "auth_own_read" on public.destination_trials
  for select to authenticated
  using (auth.uid() = user_id);

create policy "auth_own_insert" on public.destination_trials
  for insert to authenticated
  with check (auth.uid() = user_id);

-- trip_day_activity_feedback
create policy "auth_own" on public.trip_day_activity_feedback
  for all to authenticated
  using (trip_id in (select id from public.trips where user_id = auth.uid()))
  with check (trip_id in (select id from public.trips where user_id = auth.uid()));
```

## Conclusion

The RLS setup is **fundamentally sound** ÔÇö all tables have RLS ON and server-side routes use service_role which bypasses user checks. The main gap is that monetization tables lack user-level read policies, which blocks direct client-side access if ever needed. The fix migration has been created and should be pushed to production.

**No tables have RLS disabled.** No dangerously permissive `USING (true)` policies exist for user data (only for shared/aggregate tables, which is intentional).
